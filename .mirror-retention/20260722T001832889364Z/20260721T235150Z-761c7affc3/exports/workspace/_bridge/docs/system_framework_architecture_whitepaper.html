1. 文档方法

本文采用“轻量 C4 + arc42 风格”的组织方式：C4 用来分层表达系统上下文、容器、组件和运行关系；arc42 用来收纳目标、约束、质量属性、风险、决策和维护视图。文字风格遵循清晰、可扫描、能指导行动的技术写作原则。

借鉴来源在本文中的落地方式
C4 模型用上下文、层级、模块和运行路径解释系统，不把所有细节塞进一张图。
arc42保留目标、约束、构建块、运行时、部署/运维、风险和技术债视角。
技术写作实践用明确标题、短段落、表格和状态标记提升可读性。
架构中心实践把架构说明与验证、运行、风险、决策联系起来，而不是只画静态结构。

这不是新的治理系统。它只把已有事实、入口和规则重排成更容易理解的架构文档。

3. 设计原则

根因优先
先区分配置、协议、会话绑定、当前调用和业务结果，不用表面成功或失败替代判断。
Native MCP 优先
原生 MCP 是首选执行路径；Hub 和 fallback 只在当前 turn 不可用或有明确归属时使用。
安全边界不下放
fallback 不能绕过原工具权限；令牌只能放开狭窄能力，不能变成管理员身份。
修改可回滚
文件改动前走备份路由，历史散落备份通过 repair-plan 治理。
验证可机器读取
系统级模块必须有 snapshot、doctor、repair-plan、validate 或等价接口。
收口必须显性
需要写记忆、技能、基线或工具矩阵时，先展示提案，再按批准执行。

4. 标准工作机制

一次非简单任务的理想路径是“分类、召回、选工具、执行、验证、收口”。这条路径的价值在于减少随手命令，避免重复踩坑，也避免治理本身失控。

5. 工具与 MCP 机制

工具层的核心问题不是“有没有配置”，而是“当前 turn 是否真的可调用”。因此 MCP 健康必须分层看，不能把协议 smoke 成功冒充当前调用成功。

图 3：MCP 可用性五层证据
config_ok
配置存在，codex mcp list 合理。
protocol_ok
initialize / tools-list smoke 成功。
current_turn_exposed
当前 Codex turn 暴露对应 namespace。
current_turn_callable
当前 turn 真正完成一次工具调用。
call_completed
目标调用返回有效业务结果。
情况判断动作
配置和协议都好，但当前 turn 没 namespacesession/tool-surface drift记录负证据；不误判为服务坏。
调用返回 Transport closed当前 native handle 已不可用本 turn 熔断该 handle；走同权限 fallback 或 Hub；保留负证据。
Hub 可用，Native 不可用连续性路径可用只在权限等价时兜底，不冒充原生 MCP 成功。
Native 调用成功当前 turn 正证据收口时写入工具正证据，帮助下次判断。

8. 微信桥接机制

微信桥接负责把手机端消息转成 Codex 可处理任务，再把结果按 owned-result marker 回发。它的关键不是“能发出去一次”，而是权限、所有权、回执、附件和失败恢复都可解释。

图 5：微信桥接运行链路

权限
账号身份风险等级临时令牌附件区隔离
投递
codex_delegationack 收到owned-result 完成
恢复
repair队列状态回执验证失败分类

文本回发账号附件区发送确认

约束：mobile_ack 只表示收到，不表示完成。结果必须使用当前任务的 exact begin/end marker。临时令牌只放开指定能力，不能读取敏感信息、破坏本机或扩大到管理员权限。

9. 邮箱与统一调度

邮箱模块已经从“发件脚本”演进为发件、收件、回信、统一调度和任务归档的组合系统。收件箱通过主题中的“待处理”释放信号，让 worker 生成回信任务；真正发送仍走正常 outbox 流程。

这里的优先级不是为了让某些邮件特权插队，而是为了避免系统卡顿：附件、高风险、正文异常或未知发件人进入 review；低风险回信进入统一队列；普通入站只入库不触发 worker。

10. 文件、备份和资源库

文件修改必须先问“是否已有规划备份路径”。有项目备份路由就用项目路由；没有就进入统一备份库。备份不是简单复制文件，而是要带 manifest、hash、备注、来源、目的和回滚说明。

11. 记录库治理

执行记录、维护记录和资源库文档已经达到需要索引化管理的规模。正确做法不是广扫和人工打开大文件，而是用派生索引、query、metrics 和 doctor。

命令用途
python _bridge\shared\record_store_maintenance.py snapshot查看记录根、索引状态和关键容量。
python _bridge\shared\record_store_maintenance.py doctor识别索引、体量、路径漂移和治理风险。
python _bridge\shared\record_store_maintenance.py repair-plan生成修复计划，默认不直接执行。
python _bridge\shared\record_store_maintenance.py validate验证入口和索引状态是否可用。
python _bridge\shared\record_store_maintenance.py metrics输出机器可读指标。

当前统一入口已经修正为 _bridge/shared/record_store_maintenance.py，不再使用旧的漂移路径。

12. 维护面和验证

系统级模块必须能回答四个问题：现在是什么状态，哪里有风险，如何安全修复，修复后如何验证。缺少这四个问题中的任何一个，系统都会变成“只能靠人盯着”的脆弱状态。

14. 后续路线

  1. 工具层：保持 Native MCP 优先，同时让 Hub 成为同权限、可审计、不断路的 fallback；每次调用区分 current-turn callable 和 protocol smoke。
  2. 记忆层：note/PMB/画像/外部知识统一走整理计划和审批清单，避免静默写入，也避免可复用经验丢失。
  3. 桥接层：继续收紧 owned-result、令牌口令、附件区和回执确认，避免重复回发、误判失败和越权生成物。
  4. 记录层：继续索引化、间隔维护和受控冷归档，减少大文件直接查询。
  5. 文档层：保持本文为人类总览，工具矩阵和维护面地图仍是执行事实来源；文档变化必须读回和验证。

一句话总结：这套系统的正确姿势是：先用规则和记忆确定方向，再用技能和工具执行，最后用维护面验证，并把可复用经验以审批形式沉淀。
